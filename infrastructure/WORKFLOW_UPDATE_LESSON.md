# Урок: НЕ МЕНЯТЬ РАБОЧИЙ WORKFLOW

**Дата:** 2025-12-21  
**Проблема:** Попытка "исправить" рабочий workflow привела к изменению формата параметров

## 🎯 Суть проблемы

Я пытался "исправить" рабочий workflow `creative_decomposition_llm`, изменив формат `tableId` с простой строки на объект, хотя workflow уже работал правильно.

## ❌ Что я делал не так

1. **Не проверял актуальный формат перед изменениями**
   - Не прочитал актуальный workflow через `n8n_get_workflow`
   - Не проверил последние executions
   - Не понял, какой формат используется

2. **Пытался "исправить" то, что уже работало**
   - Актуальный workflow использовал `"tableId": "transcripts"` (строка) ✅
   - Я пытался заменить на `"tableId": {"__rl": true, "value": "transcripts", "mode": "name"}` ❌

3. **Добавлял параметры "на всякий случай"**
   - Добавлял `resource: "row"`, `operation: "create"`, `dataToSend: "defineBelow"`
   - Эти параметры не были указаны в актуальном workflow, но он работал

4. **Опирался на правила, а не на реальность**
   - Правила в `.cursor/rules/n8n-workflow-patterns.mdc` описывали один конкретный случай
   - Я слепо применял их ко всем workflow, не проверяя актуальный формат

## ✅ Правильный подход

### Перед ЛЮБЫМИ изменениями существующего workflow:

1. **Прочитать актуальный workflow:**
   ```javascript
   n8n_get_workflow({id: "workflow-id", mode: "full"})
   ```

2. **Проверить последние executions:**
   - Проверить статус последних executions
   - Убедиться, что workflow работает

3. **Понять актуальный формат:**
   - Какой формат `tableId` используется? (строка или объект?)
   - Какие параметры указаны? (`resource`, `operation`, `dataToSend`?)
   - Какие параметры НЕ указаны, но workflow работает?

4. **НЕ МЕНЯТЬ то, что работает:**
   - Если `tableId` - строка и workflow работает - НЕ МЕНЯТЬ на объект
   - Если `tableId` - объект и workflow работает - НЕ МЕНЯТЬ на строку
   - Если параметры не указаны, но workflow работает - НЕ ДОБАВЛЯТЬ их

5. **Менять только при ошибках:**
   - Если workflow НЕ работает - тогда исправлять
   - Если есть ошибки в executions - тогда исправлять
   - Если пользователь явно просит изменить - тогда менять

## 📋 Разница между операциями

### `getAll` операция (в `creative_ingestion_webhook`):
```json
{
  "operation": "getAll",
  "tableId": {
    "__rl": true,
    "value": "creatives",
    "mode": "name"
  },
  "filters": {...}
}
```
- **Требует объектный формат** для `tableId`
- Используется для фильтрации

### `create` операция (в `creative_decomposition_llm`):
```json
{
  "useCustomSchema": true,
  "schema": "genomai",
  "tableId": "transcripts",  // Простая строка - РАБОТАЕТ!
  "fieldsUi": {...}
}
```
- **Может работать с простым строковым форматом**
- n8n автоматически определяет операцию по наличию `fieldsUi`

## 🔍 Почему я ошибался

### Источник ошибки:

Я опирался на **правила из `.cursor/rules/n8n-workflow-patterns.mdc`**, которые говорили:

> "Всегда использовать объектный формат для `tableId`: `{"__rl": true, "value": "table_name", "mode": "name"}`"

Но эти правила были написаны на основе **другого workflow** (`creative_ingestion_webhook`), который использует объектный формат для `getAll` операций.

### Проблема правил:

Правила не были универсальными - они описывали один конкретный случай, но я применял их ко всем workflow без проверки актуального формата.

## ✅ Что исправлено

1. **Обновлены правила** в `.cursor/rules/n8n-workflow-patterns.mdc`:
   - Добавлен раздел "КРИТИЧЕСКИ ВАЖНО: НЕ МЕНЯТЬ РАБОЧИЙ WORKFLOW"
   - Уточнено, что формат `tableId` зависит от операции
   - Добавлен чеклист перед изменением существующего workflow

2. **Создан документ** `WORKFLOW_DIFF_ANALYSIS.md`:
   - Детальное сравнение актуального и "исправленного" формата
   - Объяснение, почему актуальный формат правильный

3. **Создан этот документ** для фиксации урока

## 🎯 Правило на будущее

**ПЕРЕД ЛЮБЫМИ ИЗМЕНЕНИЯМИ WORKFLOW:**

1. ✅ Прочитать актуальный workflow через `n8n_get_workflow`
2. ✅ Проверить последние executions
3. ✅ Убедиться, что workflow работает
4. ✅ Понять, какой формат используется
5. ✅ **НЕ менять то, что уже работает**
