# Learning & Memory Policy
## GenomAI — Политика обучения и памяти системы

**Версия:** v1.2  
**Статус:** CANONICAL / CORE INTELLIGENCE  
**Обязателен для:** Decision Engine, ML, Learning Loop, Override Logic

---

## 1. Назначение документа

Данный документ определяет:
- как система GenomAI обучается,
- какие типы памяти существуют,
- что считается знанием,
- как знания устаревают (time decay),
- при каких условиях идеи «умирают» и могут быть «воскрешены».

Цель политики — обеспечить **долгосрочную обучаемость и устойчивость системы**  
в условиях нестационарного рекламного рынка, где креативы имеют ограниченный срок жизни.

---

## 2. Базовые принципы обучения

### 2.1 Система учится на решениях, а не на данных

- Обучение происходит **на последствиях решений**, принятых Decision Engine.
- Сырые данные (метрики, тексты, креативы) **не являются знаниями**.
- Знанием считается только цепочка:
  > решение → результат → зафиксированный исход во времени.

---

### 2.2 Рынок — единственный учитель

- Ни ML, ни LLM, ни человек не могут объявить идею «рабочей».
- Единственный валидный сигнал обучения — **реакция рынка**.
- Внутренние сигналы используются только как вспомогательные.

---

### 2.3 Время — обязательный компонент знания

- Любое знание имеет срок актуальности.
- Старые знания автоматически теряют вес.
- Отсутствие time decay считается **критической архитектурной ошибкой**.

---

## 3. Типы памяти в системе

GenomAI использует **несколько независимых типов памяти**, каждый со своей ролью.

---

### 3.1 Outcome Memory (память результатов)

**Назначение:**  
Хранит зафиксированные реакции рынка.

**Содержит:**
- hypothesis_id
- idea_id
- метрики
- контекст запуска
- временные границы
- горизонт оценки
- environment_context (опционально): account_state, platform_state, auction_pressure

**Правило:**  
Outcome Memory является фактической и не интерпретируется напрямую.

**Temporal Aggregation:**
- Learning работает только с **агрегированными outcome**
- Outcome интерпретируется только после стабилизации во времени
- Raw metrics не используются для обучения напрямую

**Environment Context Weighting:**
- Outcome updates memory relative to environment context
- Плохой outcome в degraded environment:
  - ❌ не убивает идею
  - ❌ не снижает confidence резко
- Хороший outcome в degraded environment:
  - ✅ повышает confidence сильнее обычного
- **Это не нормализация, это мягкое весовое смещение.**

**Origin Type & Learning Rules:**
- `system` outcome (origin_type = system) → **causal updates**:
  - обновление confidence (с учётом environment weighting)
  - обновление fatigue levels
  - обновление death memory
  - обновление cluster priors
  - обновление transition effectiveness
- `user` outcome (origin_type = user) → **observational memory**:
  - расширение пространства идей (Idea Registry)
  - обновление similarity signals
  - **НЕ обновляет** confidence, fatigue, death memory, cluster priors
  - **НЕ используется** Decision Engine для принятия решений

---

### 3.2 Decision Memory (память решений)

**Назначение:**  
Хранит историю решений Decision Engine.

**Содержит:**
- тип решения
- причины принятия
- пройденные и проваленные проверки
- состояние системы на момент решения

Используется для:
- аудита,
- дебага,
- анализа политики решений.

---

### 3.3 Death Memory (память смерти идей)

**Назначение:**  
Хранит идеи и кластеры, запрещённые к повторению.

**Причины попадания:**
- устойчивый провал,
- повторный провал после resurrection,
- стратегический запрет.

**Состояния:**
- SOFT_DEAD — временный запрет,
- HARD_DEAD — устойчивый запрет,
- PERMANENT_DEAD — запрещено до override.

Death Memory **не подвержена time decay автоматически**.

---

### 3.4 Fatigue Memory (память выгорания)

**Назначение:**  
Отслеживает усталость аудитории от повторяющихся идей.

**Уровни:**
- skin — усталость от исполнения (форма подачи, тон, стиль речи в транскрипте)
- message — усталость от формулировок/скрипта
- angle — усталость от смыслового угла
- transition — усталость от повторяющегося перехода состояния
- tension — усталость от одинаковой формы конфликта
- context — усталость от рамки доверия
- exhausted — кластер временно непригоден

**Важно:**
- Fatigue ≠ Failure,
- Fatigue влияет на допустимость идей, а не на их ценность.
- **Tension fatigue ≠ idea death** — форма конфликта может выгореть, но идея остаётся живой (можно сменить tension_type)
- Fatigue отслеживается на всех уровнях независимо

---

### 3.5 Confidence Memory (память уверенности)

**Назначение:**  
Хранит степень уверенности системы в собственных убеждениях.

Используется для:
- управления exploit / explore,
- запуска Epistemic Shock,
- предотвращения самоуверенности системы.

---

## 4. Горизонты обучения

### 4.1 T1 — Ранний отклик
- Используется для:
  - отсечения явно провальных идей,
  - фиксации первичной реакции.
- Не используется для:
  - окончательных выводов,
  - Death Memory (кроме extreme случаев).

---

### 4.2 T2 — Устойчивость
- Основной горизонт обучения.
- Используется для:
  - обновления confidence,
  - оценки fatigue,
  - перевода идей в SOFT_DEAD.

---

### 4.3 T3 — Стратегическое влияние
- Используется редко.
- Влияет на:
  - политику решений,
  - правила resurrection,
  - приоритеты кластеров.

---

## 5. Learning Loop (механизм обучения)

### 5.1 Когда происходит обучение

Обучение запускается:
- по завершении горизонта оценки,
- при накоплении достаточного количества данных,
- при смене состояния системы.

---

### 5.2 Что обновляется

**Для system outcome (origin_type = system) — causal updates:**
- confidence_weight идей и кластеров
- fatigue_level (включая fatigue по state transition, tension_type, context_frame)
- death_state
- historical success / failure rates
- переходы состояний (state_before → state_after) для причинного обучения
- эффективность tension_type в разных контекстах
- совместимость context_frame с переходами

**Для user outcome (origin_type = user) — observational memory:**
- расширение пространства идей (Idea Registry)
- обновление similarity signals
- **НЕ обновляет** confidence, fatigue, death memory, cluster priors

---

### 5.3 Чего система НЕ делает

- не переобучается на одном запуске,
- не делает выводы вне горизонта,
- не стирает историю,
- не «объясняет» рынок.

---

## 6. Time Decay Policy (устаревание знаний)

### 6.1 Назначение time decay

Рекламный рынок является **нестационарной средой**.  
Даже успешные креативы **теряют эффективность примерно за один месяц**.

Time decay обеспечивает:
- защиту от бесконечного exploit,
- своевременное обновление идей,
- адаптацию к изменению рынка.

---

### 6.2 Базовое правило затухания

**Базовый срок жизни креатива / идеи — ~30 дней.**

Это означает:
- через ~30 дней без подтверждения идея считается устаревшей,
- через ~60 дней — почти неактуальной,
- через ~90 дней — не учитывается в принятии решений.

---

### 6.3 Что подвержено time decay

Time decay применяется к:
- confidence_weight идеи,
- confidence_weight кластера,
- historical_winrate,
- prior-политикам Decision Engine.

**Не подвержены time decay:**
- raw Outcome (факты),
- Decision trace,
- Death Memory.

---

### 6.4 Поведенческая модель decay (обязательные ориентиры)

- День 0: confidence_weight = 1.0  
- ~30 дней: confidence_weight ≈ 0.3–0.4  
- ~60 дней: confidence_weight ≈ 0.1  
- ~90 дней: confidence_weight ≈ 0.0  

Точная формула выбирается на этапе реализации,  
но **поведение обязано соответствовать этим ориентирам**.

---

### 6.5 Сброс и замедление decay

Time decay замедляется или частично сбрасывается, если:
- идея получает подтверждение рынка в горизонте T2,
- идея адаптирована в новый кластер,
- зафиксирован стабильный рыночный режим.

В этом случае:
- обновляется timestamp последнего подтверждения,
- confidence_weight частично восстанавливается.

---

### 6.6 Ускоренный decay

Decay ускоряется, если:
- зафиксирован fatigue уровня `message` или выше,
- идея неоднократно отклоняется Decision Engine,
- система находится в режиме Recovery или Regime Transition.

---

### 6.7 Связь time decay и Decision Engine

Decision Engine:
- использует confidence **с учётом decay**,
- не доверяет историческим виннерам без свежего подтверждения.

Старая идея может быть отклонена,
даже если ранее показывала высокий результат.

---

### 6.8 Time decay и resurrection

При resurrection:
- confidence_weight устанавливается на 0.1–0.2,
- идея запускается в изолированном контуре,
- применяется усиленный мониторинг T1/T2.

Resurrection **никогда не восстанавливает confidence полностью**.

---

### 6.9 Запрещённые действия

Запрещено:
- отключать time decay,
- вручную замораживать confidence,
- считать старые успехи равными новым,
- использовать historical_winrate без decay.

---

### 6.10 Финальное правило time decay

> **Каждая идея имеет срок жизни.  
> Даже лучшие идеи умирают без подтверждения рынка.**

---

## 7. Роль человека в обучении

Человек:
- не обучает систему напрямую,
- не помечает идеи как «хорошие»,
- не удаляет память.

Человек может:
- объявить смену режима,
- инициировать resurrection,
- зафиксировать внешний фактор.

---

## 8. Epistemic Shock Policy (анти-зацикливание)

### 8.1 Назначение

Epistemic Shock — обязательный механизм предотвращения зацикливания системы на ложных убеждениях.

### 8.2 Rate Limiting (критично)

**Проблема:**  
Без rate-limit Shock может стать либо слишком редким (никогда не срабатывает), либо слишком частым (ломает exploit).

**Правила:**
- **Максимальная частота:** не чаще 1 shock на N решений (например, 1 на 50–100 решений)
- **Минимальный cooldown:** между shock-решениями должен быть обязательный интервал (например, минимум 24 часа)
- **Бюджет:** выделенный бюджет на shock (например, не более 5–10% от общего бюджета)

**Результат:**
- Shock остаётся инструментом, а не режимом
- Система не «дергается»
- Exploit не нарушается чрезмерным тестированием

### 8.3 Триггеры Shock

Shock запускается при:
- высокой уверенности системы (confidence_bucket = strong)
- низком разнообразии кластеров
- длительном exploit без открытия нового

### 8.4 Изоляция

- Shock идеи запускаются в изолированном контуре
- Результаты учитываются осторожно
- Shock не влияет на основной exploit-бюджет

---

## 9. Защита от деградации обучения

- Anti-confirmation (Epistemic Shock с rate limiting)
- Контроль разнообразия
- Trust-флаги данных
- Минимальные пороги данных
- Cross-horizon leakage prevention (см. раздел 9.1)

### 9.1 Cross-Horizon Leakage Prevention

**Проблема:**  
Сигналы из T1 могут случайно влиять на T3-стратегические priors.

**Правило:**  
Сигналы из T1 НЕ обновляют стратегические priors (T3).

**Детали:**
- T1 сигналы используются только для раннего отсечения явного мусора
- T2 сигналы обновляют основную политику (confidence, fatigue)
- T3 сигналы влияют на стратегические правила и resurrection policies
- Нет ретроактивного влияния ранних сигналов на долгосрочные решения

**Результат:**  
Каждый горизонт изолирован от других. Нет «загрязнения» стратегических решений ранними шумами.

---

## 10. Historical vs Active Cluster Tracking

**Проблема:**  
`active_cluster_id` может меняться со временем, но память (fatigue, death) привязана к кластеру. Нужно избежать ретроактивного «переписывания истории».

**Решение:**
- В Decision entity хранится: `cluster_at_decision` (исторический снимок)
- В Outcome entity хранится: `cluster_id_at_outcome` (исторический снимок)
- `active_cluster_id` в Idea — текущее состояние (может меняться)
- Learning Loop использует исторические cluster_id для корректного обучения во времени

**Результат:**
- Нет ретроактивного «переписывания истории»
- Learning Loop становится корректным во времени
- Fatigue и death memory привязаны к историческим кластерам

---

## 11. Изменения политики

Любые изменения:
- версионируются,
- проходят архитектурное ревью,
- не применяются задним числом.

---

## 12. Финальное правило

> **Система должна помнить свои ошибки дольше,  
> чем свои успехи.**

Это фундамент долгосрочной выживаемости GenomAI.
