# User Flows Specification
## GenomAI — Canonical User Flows (Layer 2, updated)

**Версия:** v1.1  
**Статус:** CANONICAL / LAYER 2  
**Приоритет:** Высокий  
**Основан на:** Usage Doctrine, TELEGRAM_INTERACTION_MODEL, INPUT_NORMALIZATION, DATA_FLOW

---

## 1. Purpose

Документ описывает реальные пользовательские сценарии работы медиабайера с системой GenomAI с учётом автоматической транскрипции и трекинга через tracker_id.

Документ:
- описывает последовательности событий, а не UI,
- отражает push-модель,
- не описывает реализацию и инфраструктуру.

---

## 2. Core User Role

### Media Buyer

**Медиабайер:**
- передаёт факты внешнего мира (ссылки, метрики),
- запускает креативы вне системы,
- получает решения и новые транскрипты как результат работы системы.

**Он:**
- не управляет логикой,
- не взаимодействует с Decision Engine,
- не загружает транскрипты вручную.

---

## 3. Flow 1 — Creative Registration (Primary Entry Flow)

### 3.1 Цель

Зарегистрировать новый креатив в системе.

### 3.2 Вход от пользователя

- `video_url` — ссылка на видео-креатив
- `tracker_id` — идентификатор из системы трекинга (Keitaro)

**Важно:**  
- Это **user-generated creative** (внешний креатив медиабайера)
- Не связан с Hypothesis системы
- Используется для расширения пространства идей (observational learning)

### 3.3 Последовательность

1. **Медиабайер отправляет в Telegram:**
   - ссылку на видео
   - tracker_id

2. **Система регистрирует Creative reference**
   - создаётся Creative с video_url и tracker_id
   - Creative помечается как `pending_transcription`

3. **Запускается автоматический transcription workflow**
   - система извлекает **ТОЛЬКО аудиочасть** (текст того, что говорит лицо в ролике)
   - визуалы, изображения, primary/description не обрабатываются

4. **При успешной транскрипции:**
   - выполняется decomposition (LLM-assisted)
   - создаётся или обновляется Idea
   - вызывается Decision Engine

5. **Пользователь асинхронно получает:**
   - результат решения (Decision Result Payload)
   - (опционально) новые транскрипты для теста (Generated Transcript Payload)

### 3.4 Ключевые правила

- пользователь **не ждёт** мгновенного ответа
- отсутствие ответа = нормальное состояние
- транскрипт пользователь **никогда не видит** и **не редактирует**

---

## 4. Flow 2 — Transcription Failure

### 4.1 Цель

Корректно обработать невозможность транскрипции.

### 4.2 Последовательность

1. Видео недоступно / ошибка ASR
2. Creative помечается как `invalid`
3. Пользователь получает уведомление об отказе (Error/Rejection Payload)
4. Никакие сущности Layer 1 не создаются
5. Состояние системы не изменяется

---

## 5. Flow 3 — Receiving New Test Transcripts

### 5.1 Цель

Получить новые транскрипты для тестирования.

### 5.2 Последовательность

1. Decision Engine разрешает Idea (APPROVE / ALLOW_WITH_CONSTRAINTS)
2. Hypothesis Factory генерирует 1+ транскриптов
3. Система отправляет пользователю:
   - транскрипты (Generated Transcript Payload)
   - пометку «для теста»
   - связь с исходным tracker_id (неявно)

**Важно:**  
- Это **system-generated creative** (сгенерированный системой)
- Связан с Hypothesis
- При получении performance → `origin_type = system` → causal updates в Learning Loop

**Критическое ограничение:**  
Транскрипты содержат **ТОЛЬКО текст того, что говорит лицо в ролике**.  
**НЕ содержат:** визуалы, изображения, primary/description.

### 5.3 Ключевые правила

- транскрипты **не ранжируются**
- нет пометок «лучше / хуже»
- пользователь **не запрашивает** генерацию
- транскрипты не являются «улучшенной версией старого»

---

## 6. Flow 4 — External Launch

### 6.1 Цель

Запуск креативов во внешней рекламной системе.

### 6.2 Последовательность

1. Медиабайер вручную запускает креативы (вне системы GenomAI)
2. Система GenomAI:
   - не участвует
   - не оптимизирует
   - не наблюдает процесс

**Execution — внешняя граница системы.**

---

## 7. Flow 5 — Performance Feedback Submission

### 7.1 Цель

Передать системе фактическую реакцию рынка.

### 7.2 Вход от пользователя

- `tracker_id`
- performance-факты (spend, metric, time window)
- environment signals (опционально): account_state, platform_state, auction_pressure

### 7.3 Последовательность

1. Пользователь отправляет performance-данные (с опциональными environment signals)
2. Система валидирует Outcome (Data Contracts)
3. Система определяет `origin_type`:
   - если `tracker_id` связан с Hypothesis → `origin_type = system`
   - если `tracker_id` не связан с Hypothesis → `origin_type = user`
4. Outcome связывается с Creative через tracker_id
5. Outcome помечается как `trusted` / `untrusted`
6. Learning Loop обновляет память (если условия выполнены):
   - `system` outcome → causal updates (confidence, fatigue, death memory) с учётом environment
   - `user` outcome → observational memory (расширение пространства идей, без causal updates)
7. Пользователь получает Outcome Acknowledgement Payload

### 7.4 Ключевые правила

- только факты
- без интерпретаций
- обучение не гарантировано на каждом Outcome
- система не отправляет интерпретаций или оценок
- environment может быть указан, но:
  - не гарантирует обучение
  - не отменяет решения Decision Engine
  - используется только для корректной интерпретации Outcome в Learning Loop
- **разные последствия для обучения:**
  - `system` outcome → обновляет confidence, fatigue, death memory
  - `user` outcome → расширяет пространство идей, не обновляет causal память

---

## 8. Flow 6 — Re-evaluation Without User Action

### 8.1 Цель

Повторная оценка идей без новых пользовательских входов.

### 8.2 Сценарии

- накопление Outcome
- смена режима системы
- time decay
- обновление cluster priors
- изменение fatigue levels

### 8.3 Поведение

**Система может:**
- начать снова выдавать транскрипты (если Idea стала допустимой)
- замолчать (если решение defer)
- перевести идею в death (через Death Memory)

**Пользователь не инициирует этот процесс.**

---

## 9. Flow 7 — Deferred / Rejected Idea

### 9.1 Цель

Обработка отказа системы.

### 9.2 Последовательность

1. Decision Engine возвращает REJECT / DEFER
2. Пользователь получает уведомление (Decision Result Payload)
3. Никаких действий не требуется от пользователя
4. Отказ ≠ ошибка

### 9.3 Содержание уведомления

- тип решения (reject / defer)
- краткое описание причин (из Decision Trace)
- **не содержит:** scores, confidence, fatigue, внутренние метрики

---

## 10. Flow 8 — Silent Operation Mode

### 10.1 Цель

Зафиксировать нормальность молчания системы.

### 10.2 Если:

- пользователь отправил ссылку
- и не получил ответа

### 10.3 Это означает:

- система в обработке
- либо решение defer
- либо нет допустимых выходов

**Молчание — валидное состояние системы.**

---

## 11. Forbidden User Flows

**Запрещено:**
- отправлять транскрипты вручную
- редактировать текст
- запрашивать генерацию
- выбирать идеи
- повторно слать performance для «переубеждения»
- передавать performance без tracker_id
- отправлять визуалы, изображения, primary/description
- запрашивать объяснения решений
- просматривать внутренние состояния (memory, fatigue, scores)

**Любая из этих попыток — некорректное использование продукта.**

---

## 12. Temporal Characteristics

- все флоу **асинхронны**
- один вход → 0..N выходов
- порядок сообщений **не гарантирован**
- система не ожидает Outcome для продолжения работы

---

## 13. Final Rule

**Пользователь взаимодействует с системой через ссылки и факты, а все смысловые и текстовые преобразования являются внутренним делом GenomAI.**
