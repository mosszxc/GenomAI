# Telegram Interaction Model Specification
## GenomAI — Telegram Interaction Model (Layer 2)

**Версия:** v1.0  
**Статус:** CANONICAL / LAYER 2  
**Приоритет:** Высокий  
**Основан на:** Usage Doctrine, DATA_FLOW, ENTITY_LIFECYCLE, SYSTEM_ARCHITECTURE

---

## 1. Purpose

Данный документ описывает модель взаимодействия пользователя с системой GenomAI через Telegram-бота.

Документ:
- фиксирует типы сообщений и событий,
- определяет push-модель коммуникации,
- описывает допустимые пользовательские действия,
- не описывает UI, кнопки, API, реализацию, n8n, базы данных.

**Telegram рассматривается как транспортный канал, а не логический компонент системы.**

---

## 2. Core Interaction Principle

**Telegram-бот не является интерфейсом управления системой.**  
**Он является каналом доставки фактов и результатов.**

**Следствия:**
- пользователь не запрашивает решения,
- пользователь не управляет логикой,
- бот не ведёт диалог,
- система сама инициирует ответы.

---

## 3. Interaction Model Type

### 3.1 Push-Only Model

Коммуникация строго событийная:

```
User sends fact
→ System processes internally
→ System sends result when готово
```

**Запросно-ответная модель архитектурно запрещена.**

### 3.2 Stateless Messaging

- каждое сообщение трактуется изолированно,
- бот не хранит контекст диалога,
- состояние существует только внутри системы, не в чате.

---

## 4. Actor Roles in Telegram

### 4.1 Media Buyer (User)

**Через Telegram пользователь может:**
- передать входные данные (транскрипты видео, performance-факты),
- получить результаты обработки (решения, новые транскрипты).

**Пользователь не может:**
- влиять на решения,
- выбирать идеи,
- управлять состоянием системы.

**Критическое ограничение:**  
Пользователь передаёт **ТОЛЬКО транскрипты видео** (текст того, что говорит лицо в ролике).  
**НЕ передаются:** визуалы, изображения, primary/description в Facebook Ads.

### 4.2 Telegram Bot (Transport)

**Бот:**
- принимает сообщения,
- передаёт их в Ingestion Layer,
- доставляет system outputs пользователю.

**Бот не:**
- интерпретирует,
- решает,
- обучается,
- хранит бизнес-логику.

---

## 5. Supported Message Categories (High-Level)

Telegram-бот поддерживает ограниченный набор категорий сообщений.

### 5.1 Inbound (User → System)

#### Creative Transcript Message
- транскрипт видео/креатива
- plain text
- без разметки и команд
- **ТОЛЬКО текст того, что говорит лицо в ролике**

#### Performance Outcome Message
- фактические результаты запуска
- числовые значения (spend, primary_metric)
- временной контекст
- контекст запуска (гео, вертикаль, источник)

#### Metadata / Context Message
- опциональный контекст запуска
- источник, вертикаль, гео и т.п.
- не влияет на Decision Engine напрямую

### 5.2 Outbound (System → User)

#### Decision Result Message
- факт принятого решения
- тип решения (approve / reject / defer / allow_with_constraints)
- краткое резюме причин (из Decision Trace)
- не содержит интерпретаций или объяснений

#### Generated Transcript Message
- новые транскрипты для теста
- всегда привязаны к решённой Idea
- **ТОЛЬКО транскрипт, не визуалы/primary/description**
- не являются «улучшенной версией старого»

#### System Notification
- ошибки валидации
- невозможность обработки
- статусные уведомления
- не содержит внутренних состояний системы

---

## 6. Message Semantics (не формат)

### 6.1 Creative Transcript

**Считается:**
- фактом внешнего мира,
- неизменяемым артефактом.

**Любой текст:**
- не интерпретируется в Telegram,
- не «разбирается» ботом,
- передаётся целиком в систему (Ingestion → Validation → Decomposition).

**Критическое ограничение:**  
Транскрипт содержит **ТОЛЬКО текст речи**, не визуальные элементы, не primary/description.

### 6.2 Performance Outcome

**Считается:**
- фактом реакции рынка,
- не интерпретацией,
- не оценкой качества.

**Если данные неполные или некорректные:**
- система уведомляет через System Notification,
- обучение не происходит,
- состояние системы не изменяется.

---

## 7. Ordering & Asynchrony

- входные сообщения не гарантируют порядок обработки,
- ответы системы могут приходить:
  - позже,
  - в другом порядке,
  - отдельными сообщениями.

**Telegram-бот не гарантирует синхронный отклик.**

**Правило:**  
Система обрабатывает сообщения асинхронно. Decision Engine может принимать новые решения до получения Outcome по предыдущим.

---

## 8. Error & Rejection Handling

**При ошибках:**
- бот отправляет явное уведомление (System Notification),
- система состояние не изменяет,
- обучение не происходит.

**Примеры причин:**
- невалидный формат (Validation Layer rejection),
- невозможность decomposition (LLM failure или schema violation),
- конфликт контекста (Data Contract violation).

**Fail-safe стратегия:**  
Лучше не принять решение, чем принять неверное.

---

## 9. Forbidden Interaction Patterns

**Архитектурно запрещено:**
- диалог с ботом (уточняющие вопросы, follow-up)
- уточняющие вопросы от бота
- команды вида «сделай лучше», «оптимизируй текст»
- выбор идей пользователем
- ручной триггер Decision Engine
- просмотр внутренних состояний (memory, fatigue, death)
- запросы «почему отклонили», «что не так»
- интерактивные кнопки для управления логикой

**Любая из этих попыток — некорректное использование продукта.**

---

## 10. Security & Authority Boundary

**Telegram:**
- не является trusted source логики,
- не определяет авторитет пользователя,
- не даёт привилегий.

**Все authority-решения существуют вне Telegram.**

**Human Override** (если требуется) происходит через отдельный механизм, не через Telegram-бота.

---

## 11. Evolution Rule

**Изменение Telegram-модели допустимо, если:**
- не нарушает Usage Doctrine (Layer 0),
- не вводит запросную модель,
- не раскрывает Decision Engine,
- не превращает бота в интерфейс управления логикой.

**Любое изменение требует:**
- новой версии документа,
- обратной совместимости по смыслу,
- проверки на соответствие Layer 0–1.

---

## 12. Layer Boundary Notice

Этот документ:
- принадлежит Layer 2 (Product & Interaction),
- опирается на Layer 0–1,
- не влияет на Decision Engine, Schema или Learning.

---

## 13. Final Rule

**Если Telegram начинает "управлять" системой — архитектура считается нарушенной.**

**Telegram — транспорт, а не мозг.**
