# Architecture Lock / Doctrine
## Autonomous Creative Decision System (GenomAI)

## Версия
v1.0

## Статус
ARCHITECTURE LOCKED  
Изменения возможны только через осознанное архитектурное решение.

---

## 1. Назначение документа

Данный документ фиксирует **архитектурные инварианты, запреты и роли компонентов** системы GenomAI.

Цель:
- предотвратить архитектурную деградацию,
- исключить неконтролируемое использование ИИ,
- обеспечить воспроизводимость решений,
- сохранить обучаемость системы на длинной дистанции.

Этот документ **выше PRD, задач, спринтов и конкретных реализаций**.

---

## 2. Определение системы

**GenomAI — это система принятия решений о допустимости креативных идей**, а не генератор текстов и не оптимизатор объявлений.

Система:
- решает, *какие идеи можно тестировать*,
- управляет риском, новизной и выгоранием,
- обучается на последствиях собственных решений,
- минимизирует повторение стратегических ошибок.

---

## 3. Основные архитектурные принципы

### 3.1 Рынок — единственный источник истины
- Все гипотезы проверяются только рынком.
- Внутренние модели, ИИ и эвристики не являются арбитром истины.
- Любая логика, игнорирующая рыночную обратную связь, считается дефектной.

---

### 3.2 Решения должны быть детерминированными
- Принятие решений должно быть воспроизводимым.
- Одинаковые входные данные → одинаковое решение.
- Любое решение должно иметь трассировку причин (decision trace).

---

### 3.3 Идея ≠ Текст ≠ Креатив
- **Идея** — смысловая гипотеза.
- **Текст / видео** — реализация идеи.
- **Креатив** — носитель реализации.

Смешение этих уровней запрещено.

---

### 3.4 Schema-first архитектура
- Все данные описываются через canonical schema.
- Если переменной нет в schema — её не существует.
- Бизнес-логика не может опираться на неописанные поля.

---

### 3.5 Environment Separation
- Система различает эффект идеи и эффект среды исполнения.
- Факторы среды (account state, platform state, auction pressure) не являются основанием для принятия решений Decision Engine.
- Environment используется только для корректной интерпретации Outcome в Learning Loop.
- **Environment не является escape hatch для оправдания плохих решений.**

---

### 3.6 Decision-Driven Learning vs Observational Learning
- Система принимает решения только на результатах собственных решений (Decision-driven learning).
- Внешние креативы используются исключительно для расширения пространства идей (Observational learning).
- **Запрет принимать решения без причинной цепочки:** Decision Engine использует только causal channel (результаты собственных решений).
- Learning Loop читает два источника: system-outcome (causal updates) и user-outcome (observational memory).
- Система никогда не превращается в spy-сервис или корреляционный ML.

---

## 4. Роли компонентов системы (жёстко зафиксированы)

### 4.1 Decision Engine (ядро системы)
- Принимает решения о допустимости идей.
- Основан на правилах (if / then).
- Использует сигналы, но не объяснения.

**Запрещено:**
- использовать LLM для принятия решений,
- принимать решения без фиксированных правил,
- скрывать логику принятия решения.

---

### 4.2 ML / Data Science слой
- Используется как сенсор и аналитический помощник.
- Предоставляет:
  - similarity score (raw float),
  - novelty score (raw float),
  - confidence weight (raw float),
  - сигналы выгорания,
  - статистические индикаторы.

**Критическое правило дискретизации:**
- ML предоставляет raw float значения
- Decision Engine получает ТОЛЬКО bucket-значения:
  - `novelty_bucket`: low/medium/high
  - `similarity_bucket`: low/medium/high
  - `confidence_bucket`: weak/normal/strong
- Конвертация float → bucket явная, версионированная, воспроизводимая

**Запрещено:**
- автоматически approve/reject идеи,
- bypass'ить Decision Engine,
- обучаться end-to-end на прибыли без ограничений,
- передавать raw float в Decision Engine (только buckets),
- скрывать логику в непрерывных значениях.

---

### 4.3 LLM (языковые модели)
- Используются строго для:
  - декомпозиции транскриптов видео (только текст речи),
  - классификации по schema,
  - генерации вариантов реализации.

**Критическое ограничение:**  
LLM обрабатывает **ТОЛЬКО транскрипт видео** — текст того, что говорит лицо в ролике.  
**НЕ обрабатываются:** визуалы, изображения, primary/description в Facebook Ads.

**Запрещено:**
- принимать стратегические решения,
- выбирать идеи для теста,
- менять schema,
- добавлять новые переменные.

LLM не является источником истины.

---

### 4.4 n8n / Orchestration слой
- Используется для:
  - автоматизации,
  - маршрутизации данных,
  - интеграций,
  - триггеров процессов.

**Запрещено:**
- размещать бизнес-логику принятия решений,
- делать n8n местом, где "думает система".

n8n — нервная система, не мозг.

---

### 4.5 Human Override (человек)
- Человек не участвует в ежедневном цикле решений.
- Человек может действовать только как interrupt.

Допустимые override:
- Regime Declaration (смена режима рынка),
- Cultural Shift (изменение внешнего контекста),
- Strategic Bet (осознанный риск),
- External Constraint (платформенные ограничения).

Все override логируются и не ломают learning loop.

---

## 5. Архитектурные запреты (non-negotiable)

Запрещено в любой версии системы:

1. LLM как Decision Engine  
2. End-to-end reinforcement learning на profit  
3. Принятие решений внутри n8n  
4. Обход canonical schema  
5. Необъяснимые решения без decision trace  
6. Ручной выбор идей на регулярной основе  
7. Оценка результатов вне горизонтов  
8. Масштабирование без проверки выгорания  

Нарушение любого пункта считается **архитектурным дефектом**.

---

## 6. Горизонты и время как обязательный фактор

- Все решения оцениваются во времени.
- Используются горизонты:
  - T1 — ранняя реакция,
  - T2 — устойчивость,
  - T3 — стратегическое влияние.

Выводы вне соответствующего горизонта запрещены.

---

## 7. Обучение системы

- Система обучается на последствиях **собственных решений**, а не на сырых данных.
- Learning loop обновляет:
  - веса решений,
  - доверие к кластерам,
  - состояние выгорания,
  - память смерти гипотез.
- Старые знания подвержены time-decay.

---

## 8. Epistemic Shock (анти-зацикливание)

Система обязана:
- периодически тестировать идеи против собственных убеждений,
- иметь выделенный бюджет на странные гипотезы,
- изолировать shock от основного контура заработка.

Отсутствие Epistemic Shock считается риском деградации.

---

## 9. Приоритет выживания системы

- Краткосрочная прибыль вторична по отношению к:
  - обучаемости,
  - устойчивости,
  - способности адаптироваться.

Система должна переживать:
- выгорание,
- просадки,
- смену рыночного режима.
