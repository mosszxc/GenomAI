# Autonomous Creative Decision System
# Final Architecture & Decision Doctrine

## 0. Статус документа

Данный документ фиксирует окончательную архитектуру, философию и операционные принципы GenomAI — полностью автономной системы принятия креативных решений в нестабильной рыночной среде.

Документ:
- не описывает SaaS или продукт,
- не предполагает активное участие человека,
- не содержит roadmap'ов.

Это доктрина функционирования системы, а не план развития.

## 1. Определение системы

GenomAI — это автономная система принятия креативных решений, предназначенная для:
- обнаружения устойчивых связок креативных переменных,
- их эволюции через контролируемые мутации,
- управления выгоранием аудитории,
- системного повышения hitrate решений во времени.

GenomAI функционирует как внешний стратегический мозг, где:
- рынок является единственным источником истины,
- решения оцениваются по последствиям,
- обучение происходит на результатах собственных действий.

## 2. Базовые принципы (неизменяемые)

1. **Market is the only ground truth**
2. **Survival > short-term revenue**
3. **Decision → Outcome → Learning**
4. **Selection bias признаётся и не «лечится»**
5. **Выгорание — сигнал, а не ошибка**
6. **Автоматизация предпочтительнее ручных решений**
7. **Человек — interrupt, а не участник цикла**

## 3. Роль человека

Человек не участвует в ежедневных решениях.

Допустимые роли человека:
- **Regime Declaration**
- **Cultural Shift Override**
- **Strategic Bet Override**
- **Aesthetic / Taste Stop**
- **External Signal Injection**

Все override'ы:
- редкие,
- логируемые,
- не ломают learning loop,
- не обнуляют историю системы.

## 4. Архитектура принятия решений

```
External Signals (Market / Spy / Context)
            ↓
ML Advisory Layer
(similarity, novelty, fatigue signals)
            ↓
Decision Engine
(rules, policies, constraints)
            ↓
Hypothesis Factory
            ↓
Market Execution
            ↓
Outcome Logging
            ↓
Learning Loop
            ↓
Decision Engine
```

**ML никогда не bypass'ит Decision Engine.**

## 5. Learning Loop (ядро системы)

Минимальная единица обучения:
```
Hypothesis
→ Expected Outcome
→ Actual Outcome
→ Decision Weight Update
```

Обучение происходит на уровнях:
- типов мутаций,
- комбинаций переменных,
- контекстов (GEO / placement / vertical),
- повторяемых провалов.

Результат обучения — изменение будущих решений, а не объяснение прошлого.

## 6. Decision Horizon Doctrine
### (Доктрина горизонтов решений)

Ни одно решение не существует вне временного горизонта.

### Short Horizon (T₁)
- **0–3 дня**
- цель: первичная реакция рынка
- высокий шум допустим
- используется для exploration
- T₁ не обновляет стратегические приоры.

### Mid Horizon (T₂)
- **3–14 дней**
- цель: устойчивость и повторяемость
- основной рабочий горизонт
- решения exploit / partial exploit принимаются здесь

### Long Horizon (T₃)
- **14+ дней**
- цель: обновление политик и ограничений
- используется для:
  - изменения веса классов мутаций,
  - запрета направлений,
  - пересмотра fatigue doctrine

**Принцип:** Решение считается плохим или хорошим только в рамках своего горизонта.

## 7. Hypothesis Death & Resurrection Rules
### (Правила смерти и воскрешения гипотез)

Система различает три состояния гипотез.

### 1. Soft Failure
- единичный или контекстный провал
- гипотеза остаётся доступной
- получает отрицательный вес

### 2. Hard Failure
- повторяемый провал в разных контекстах
- exploit запрещён
- допускаются только структурные мутации

### 3. Dead Hypothesis

Гипотеза признаётся мёртвой, если:
- провал повторился ≥ N раз
- в ≥ M независимых кластерах
- на горизонте ≥ T₂

**Dead Hypothesis:**
- полностью исключается из генерации
- логируется в Death Memory
- используется только как анти-пример

### Resurrection

Воскрешение мёртвых гипотез запрещено по умолчанию.

Допустимо только через:
- **External Override**
- **Cultural Shift**
- **Regime Declaration**

История провалов никогда не обнуляется.

## 8. Fatigue Doctrine
### (Управление выгоранием)

Выгорание детектируется на уровне кластеров, а не отдельных креативов.

Стадии exhaustion:
1. **Skin Exhaustion**
   → разрешены только визуальные мутации
2. **Message Exhaustion**
   → запрещено повторение message-структур
3. **Angle Exhaustion**
   → exploit полностью запрещён
4. **Forced Novelty**
   → принудительное exploration

Реакция происходит автоматически, без участия человека.

## 9. Spy Override Doctrine

Spy-креативы рассматриваются как внешние сигналы, а не готовые решения.

**Правила:**
- помечаются как `override_type = spy_import`
- проходят полную декомпозицию
- включаются в learning loop
- доля ограничена (10–30%)

**Spy используется для:**
- bootstrap новой меты,
- прыжков через выгорание,
- ускорения discovery.

**Spy не используется для прямого exploit.**

## 10. Epistemic Shock Doctrine
### (Защита от само-подтверждения)

GenomAI признаёт риск self-confirmation loop.

Для его предотвращения используется **Epistemic Shock** — намеренное нарушение приоров системы.

**Типы Epistemic Shock:**
1. **Forced Anti-Prior**
   - Запуск гипотез с низким prior и высокой novelty
2. **Conceptual Inversion**
   - Инверсия message / tone / emotional vector
3. **Contextual Displacement**
   - Перенос паттернов в нетипичные контексты

**Ограничения:**
- shock строго лимитирован,
- не участвует в exploit,
- анализируется отдельно.

**Принцип:** Если система слишком уверена — она слепнет.

## 11. Роль машинного обучения

ML используется исключительно как **advisory-слой:**
- similarity & distance
- fatigue early detection
- novelty scoring
- hypothesis ranking (prior)

**ML:**
- не предсказывает деньги
- не принимает финальных решений
- не управляет политиками

## 12. Критерии идеальности системы

GenomAI считается идеальной, если:
- hitrate растёт во времени
- exploit и explore балансируются автоматически
- выгорание локально и управляемо
- повторение провалов минимально
- человек редко вмешивается
- система автономно работает неделями

## 13. Финальный принцип

Идеальная GenomAI — это система, которая зарабатывает, когда человек ей не мешает.

Человек остаётся источником редкого сдвига.

Все остальные решения — ответственность системы.
