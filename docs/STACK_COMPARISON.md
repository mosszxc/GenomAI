# Сравнение технологий для GenomAI: n8n vs Temporal vs Альтернативы

**Дата:** 2025-01-22  
**Статус:** Анализ и рекомендация  
**Контекст:** Выбор orchestration layer для GenomAI MVP

---

## 🎯 Ключевые требования GenomAI

### Архитектурные требования:
1. **Детерминированный Decision Engine** - rules-based (IF/ELSE/SWITCH), не ML
2. **Event-driven архитектура** - через `event_log` (append-only)
3. **Scheduled tasks** - daily scans (cron: `0 3 * * *`)
4. **LLM integration** - для Decomposition и Hypothesis Factory
5. **Database operations** - Supabase (PostgreSQL) как source of truth
6. **Webhook handling** - приём Creative ingestion
7. **Learning Loop** - rules-based агрегаты с time decay
8. **Простота разработки** - быстрая итерация для MVP

### Текущие workflows:
- ✅ Creative Ingestion Webhook (работает)
- ✅ Outcome Ingestion Keitaro (создан, требует тестирования)
- ⏳ Decision Engine (детерминированный)
- ⏳ Hypothesis Factory (LLM integration)
- ⏳ Learning Loop (агрегаты)

## 🔍 Детальный анализ по технологиям

### 1. n8n (текущий выбор) ✅

**Плюсы:**
- ✅ **Визуальный редактор** - идеально для быстрой итерации MVP
- ✅ **Нативная интеграция** с Supabase, HTTP, Webhooks
- ✅ **Простота разработки** - не нужно писать код для базовых операций
- ✅ **Отладка** - визуальное выполнение, видно каждый шаг
- ✅ **Бесплатно** для MVP (self-hosted)
- ✅ **Быстрая итерация** - изменения видны сразу
- ✅ **Детерминированные workflows** - IF/ELSE/SWITCH работают отлично
- ✅ **Event logging** - легко интегрировать с `event_log`

**Минусы:**
- ⚠️ **Версионирование** - через экспорт JSON (но можно в Git)
- ⚠️ **Масштабируемость** - ограничена для очень больших нагрузок
- ⚠️ **Long-running workflows** - не идеально для очень долгих процессов
- ⚠️ **State management** - базовый (но достаточно для MVP)

**Вывод:** **Идеально для MVP** - быстрая разработка, визуальная отладка, нативные интеграции.

---

### 2. Temporal ⚠️

**Плюсы:**
- ✅ **Отличный state management** - идеально для долгих процессов
- ✅ **Гарантии выполнения** - даже при сбоях
- ✅ **Версионирование** - Git-native
- ✅ **Масштабируемость** - отлично для production
- ✅ **Retry logic** - очень гибкий

**Минусы:**
- ❌ **Нет визуального редактора** - только код (TypeScript/Python/Go)
- ⚠️ **Требует инфраструктуры** - Temporal Server + Workers
- ⚠️ **Learning curve** - нужно изучать Temporal SDK
- ⚠️ **Интеграции** - нужно писать код для Supabase, HTTP, Webhooks
- ⚠️ **Отладка** - через логи, не визуально
- ⚠️ **Overkill для MVP** - слишком сложно для текущих задач

**Вывод:** **Отлично для production масштаба**, но **overkill для MVP**. Сложнее в разработке и отладке.

---

### 3. Airflow ⚠️

**Плюсы:**
- ✅ **Мощный** - для сложных ETL pipelines
- ✅ **Много интеграций** - через operators
- ✅ **Версионирование** - Git-native
- ✅ **Масштабируемость** - отлично

**Минусы:**
- ❌ **Сложность** - требует понимания DAG концепции
- ⚠️ **Требует инфраструктуры** - Airflow Server
- ⚠️ **Интеграции** - нужно писать код для Supabase, HTTP
- ⚠️ **Webhook handling** - не нативно
- ⚠️ **Overkill для MVP** - слишком тяжеловесно

**Вывод:** **Для ETL отлично**, но **не подходит для GenomAI** - слишком сложно для текущих задач.

---

### 4. Prefect ⚠️

**Плюсы:**
- ✅ **Современный** - хороший API
- ✅ **Версионирование** - Git-native
- ✅ **Масштабируемость** - хорошо

**Минусы:**
- ❌ **Нет визуального редактора** - только код (Python)
- ⚠️ **Требует инфраструктуры** - Prefect Server
- ⚠️ **Интеграции** - нужно писать код
- ⚠️ **Learning curve** - нужно изучать Prefect API

**Вывод:** **Хорошая альтернатива**, но **не лучше n8n для MVP** - сложнее в разработке.

---

### 5. Camunda ⚠️

**Плюсы:**
- ✅ **Визуальный редактор** - BPMN диаграммы
- ✅ **Enterprise-grade** - для больших систем
- ✅ **Версионирование** - Git-native

**Минусы:**
- ❌ **Сложность** - BPMN может быть избыточным
- ⚠️ **Требует инфраструктуры** - Camunda Server
- ⚠️ **Интеграции** - нужно писать код
- ⚠️ **Overkill для MVP** - слишком enterprise

**Вывод:** **Для enterprise отлично**, но **не подходит для MVP** - слишком сложно.

---

## 🎯 Рекомендация для GenomAI

### ✅ ОСТАВИТЬ n8n для MVP

**Причины:**

1. **Быстрая итерация** - визуальный редактор позволяет быстро создавать и тестировать workflows
2. **Нативные интеграции** - Supabase, HTTP, Webhooks работают "из коробки"
3. **Простота отладки** - визуальное выполнение, видно каждый шаг
4. **Детерминированные workflows** - IF/ELSE/SWITCH работают отлично для Decision Engine
5. **Event logging** - легко интегрировать с `event_log`
6. **Бесплатно** - для MVP не нужны дополнительные затраты
7. **Уже работает** - Creative Ingestion Webhook уже реализован и работает

### ⚠️ Когда стоит рассмотреть Temporal:

**После MVP, если:**
- Нужны очень долгие workflows (несколько дней/недель)
- Нужна гарантия выполнения при любых сбоях
- Нужна очень высокая масштабируемость
- Нужна сложная state management логика

**Но для текущих задач GenomAI:**
- Decision Engine - детерминированный, быстрый
- Learning Loop - агрегаты, не долгие процессы
- Daily scans - раз в сутки, не критично
- Webhook ingestion - быстрый процесс

**Temporal будет overkill.**
